BFLOW-001: Order Lifecycle

Service: portfolio-service Business flow Status: Partial Source: StockWise SRS (Nhóm 21)

End-to-end paper-trading order flow that ties together the three order use cases — UC-002 (buy), UC-003 (sell), and UC-004 (cancel) — and the ENTITY-003 status lifecycle. An order is validated, assets are frozen, the order is persisted as PENDING, then it is either matched to FILLED (producing a transaction) or cancelled, releasing the frozen assets.

Place & Fill Sequence

sequenceDiagram actor User participant FE as Frontend participant GW as API Gateway participant PF as portfolio-service participant MD as market-service participant DB as PostgreSQL participant MQ as RabbitMQ User->>FE: Submit order (symbol, side, qty, price) FE->>GW: POST /portfolio/order (with JWT) GW->>PF: forward request + user identity PF->>PF: validate input (BR-103) PF->>MD: request daily price band MD-->>PF: floor / ceiling (BR-100) PF->>PF: check funds (BUY, BR-101) or holdings (SELL, BR-102) PF->>PF: freeze cash or lock shares PF->>DB: insert order, status = PENDING PF->>MQ: publish ORDER_CREATED PF-->>FE: 201 order_id, status = PENDING Note over PF,MQ: Matching engine processes the order asynchronously MQ-->>PF: order matched PF->>DB: status = FILLED, append transaction (ENTITY-004) PF->>MQ: publish portfolio.exchange / updated (MSG-002)

Flow Steps

#StepRule / Artifact
1User submits an order; the gateway authenticates the JWT and forwards the request with the user identity.BR-001
2The service validates input types, values, and lot size.BR-103
3The price is checked against the daily band from Market Data.BR-100
4BUY checks available cash; SELL checks available (unlocked) holdings.BR-101, BR-102
5The corresponding cash or shares are frozen to prevent double spending.UC-002
6The order is persisted as PENDING, then ORDER_CREATED is published.ENTITY-003
7On a match, the order becomes FILLED, an append-only transaction is recorded, and a portfolio-updated event fires.ENTITY-004, MSG-002
8Cancel path: a PENDING order owned by the caller can be cancelled, which releases the frozen assets and sets CANCELLED.BR-105, UC-004

Ordering & Reliability

  • The portfolio-updated event is published only after the database write commits (NFR-REL-02).
  • Transactions are append-only (FR-PORTFOLIO-07, NFR-REL-03).
  • If event publish fails after the DB write (E9), the order is accepted and retried; the system never leaves cash or holdings negative (FR-PORTFOLIO-06).
