UC-004: Cancel Order

Service: portfolio-service Status: Implemented

Use Case Specification (SRS)

AttributeDetail
NameCancel Order (Hủy lệnh)
Primary ActorUser (investor). Also Manager and Admin.
TriggerUser clicks "Cancel" next to a PENDING order in the order book.
Pre-conditions1. Valid JWT.
2. Service healthy.
3. Broker connected.
4. Store writable.
5. The target order exists and belongs to the user.
Main Flow (specified) 1. User clicks "Cancel" on a PENDING order.
2. System validates the JWT and extracts the user id.
3. System looks up the order by order_id and verifies ownership.
4. System checks the order status is PENDING (only PENDING is cancellable).
5. System requests cancellation from the matching engine via the broker (event ORDER_CANCEL_REQUESTED).
6. System updates the order status to CANCELLED and sets cancelled_at.
7. System unfreezes the locked cash or shares back to the available balance.
8. System returns HTTP 200 with the cancelled order.
Post-conditions (specified)On success: order status is CANCELLED, frozen assets are released, a cancel event is sent. On failure: nothing changes.
Required to ImplementAn Order entity with a status lifecycle (ENTITY-003), a DELETE /portfolio/order/{orderId} endpoint (API-003), and asset freeze/unfreeze handling.
Linked FRsExtends the order-management capability of FR-PORTFOLIO-03 / FR-PORTFOLIO-04. No dedicated FR-PORTFOLIO line item covers cancellation in the current SRS table.
